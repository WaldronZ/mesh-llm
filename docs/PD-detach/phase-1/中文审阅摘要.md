## 当前项目架构总结

这个项目现在不是一个简单的单 crate Rust 应用，而是一个拆分后的 workspace。

当前主入口很薄：

- `crates/mesh-llm/src/main.rs`：创建 Tokio runtime，然后调用 `mesh_llm::run_main()`。
- `crates/mesh-llm/src/lib.rs`：基本是 re-export `mesh_llm_host_runtime`。

主要业务逻辑在：

- `crates/mesh-llm-host-runtime/src/`：真正的主机运行时，包含 CLI、runtime、mesh、API、models、network、plugins、Skippy 接入。
- `crates/mesh-llm-protocol/`：协议常量、protobuf 类型、wire 编解码。
- `crates/mesh-client/`、`crates/mesh-api/`、`crates/mesh-api-ffi/`：客户端和外部 API/FFI 表面。
- `crates/skippy-*`：分层推理、stage runtime、拓扑规划、正确性验证、benchmark 等。
- `crates/model-*`：模型引用、artifact、包格式、Hugging Face 解析等。
- `crates/mesh-llm-ui/`：React/Vite Web console，最终嵌入 binary。

接管时最重要的一点：很多旧文档还写着 `crates/mesh-llm/src/api`、`src/network`、`src/inference` 这类旧路径，但当前真实路径大多已经迁到 `crates/mesh-llm-host-runtime/src/...`。遇到路径不一致时，以代码为准。

## 运行方式总结

常用运行方式：

- `mesh-llm client --auto`：作为 client 自动加入可发现 mesh。
- `mesh-llm serve`：按 `~/.mesh-llm/config.toml` 启动配置里的模型。
- `mesh-llm serve --model <model>`：启动单个模型。
- `--log-format json`：适合脚本化观察日志。
- `--headless`：只关闭内嵌 Web UI，不是安静后台模式，管理 API 仍在。

默认端口：

- `9337`：OpenAI-compatible API，例如 `/v1/models`、`/v1/chat/completions`。
- `3131`：管理 API 和 Web console，例如 `/api/status`、`/api/runtime`。

runtime 目录优先级：

1. `MESH_LLM_RUNTIME_ROOT`
2. `$XDG_RUNTIME_DIR/mesh-llm/runtime`
3. `$HOME/.mesh-llm/runtime`

停止进程优先用：

- `mesh-llm stop`
- `just stop`

只有在实例严重卡死时，才考虑 `pkill -f mesh-llm` 这类应急方式。

## 协议和 API 总结

协议层当前至少涉及三类 ALPN：

- `mesh-llm/1`：当前 protobuf 控制面协议。
- `mesh-llm/0`：shared protocol crate 里仍存在 legacy JSON 兼容代码；owner 已确认它是历史/遗留兼容代码，不是当前正式运行协议。
- `mesh-llm-control/1`：owner/config control plane。

核心 stream id 包括 gossip、tunnel、route request、peer down/leaving、plugin channel、bulk transfer、subprotocol 等。协议字段和 protobuf schema 必须保持兼容，新增字段应走 additive 方式。

当前管理 API 路由集中在：

- `crates/mesh-llm-host-runtime/src/api/routes/mod.rs`
- `crates/mesh-llm-host-runtime/src/api/routes/runtime.rs`
- `crates/mesh-llm-host-runtime/src/api/status.rs`

重要 API 面：

- `/api/status`：节点和 mesh 状态。
- `/api/models`：mesh 模型视图。
- `/api/runtime/*`：本地 runtime、stage、process、control config。
- `/api/plugins/*`：插件 manifest、tools、HTTP binding。
- `/api/objects*`：多模态对象/附件上传。
- `/v1/*`、`/models`：OpenAI 兼容表面。
- `/api/chat*`、`/api/responses*`：会被转到 OpenAI 风格上游路径。

UI 的 TypeScript 类型在 `crates/mesh-llm-ui/src/lib/api/types.ts`，但它是消费者视角，不是服务端权威 schema。服务端字段以 Rust API payload 为准。

## 配置总结

配置文件解析优先级：

1. CLI `--config <path>`
2. `MESH_LLM_CONFIG`
3. `~/.mesh-llm/config.toml`

主要配置结构：

- `version`：可选，当前只接受 `1`。
- `[gpu]`：GPU 分配模式、并行度。
- `[owner_control]`：owner control bind 和 advertise 地址。
- `[telemetry]`：遥测开关、service name、endpoint、headers、queue 等。
- `[[models]]`：模型路径/ref、mmproj、ctx size、gpu_id、cache/batch/flash attention 等。
- `[[plugin]]`：插件 name、enabled、command、args、url。

重要约束：

- `gpu.assignment = "auto"` 时不能给模型设置 `gpu_id`。
- `gpu.assignment = "pinned"` 时每个模型都必须设置非空 `gpu_id`。
- `parallel` 必须大于等于 1。
- `batch`、`ubatch` 不能为 0。
- `telemetry.prompt_shape_metrics` 目前不支持，必须保持 false。
- 内置插件不是任意外部命令，允许字段受限。

## 测试总结

测试入口仍以 `docs/design/TESTING.md` 为主，`TEST_MATRIX.md` 做了接管期的快速索引。

最低原则：

- Rust 改动：格式检查和 `cargo check -p mesh-llm`。
- API、gossip、协议、routing、serialization 改动：不能只 build，至少跑 `cargo test -p mesh-llm --lib`。
- UI 改动：跑 `just build`。
- Rust + UI 混合改动：跑 `just build`。
- cargo 命令不要并行跑。

这个 `PD-detach` 分支特别要关注：

- gossip 版本过滤逻辑。
- 可解析且低于 `v0.60.0` peer 被拒绝的行为；未知/不可解析版本会保守放行。
- transitive idle client 过滤。
- `/api/status` 中 peer 列表是否符合预期。
- mixed-version mesh 测试。

## 安全风险总结

主要安全面：

- owner identity 和 trust policy：代码已经有 keystore、node signing、verify、revoke 等能力。
- 管理 API：默认在 console port；owner-control 相关 endpoint 有 loopback caller 检查。
- telemetry：任何遥测字段变更都要考虑隐私，不能把 prompt/content 之类数据无意发出。
- artifact transfer：`MESH_LLM_ARTIFACT_TRANSFER=open` 风险高，默认应谨慎。
- 插件：要审查 command/env 注入、stapled HTTP、MCP tools、blob/object 访问、telemetry 接入。
- 凭据：测试机器 IP、密码、SSH 命令、owner key 等不能进仓库。

6. Qwen 相关文档是当前客户 runbook，还是历史 benchmark evidence。
7. 哪些 benchmark 文档可以用于对外性能陈述。
8. Jan integration 是否仍在本仓库维护范围内。

10. management API 的正式 schema 应该放在哪里维护。

## 文档和代码不一致/疑似过期的地方

这些是 `risk-register.md` 里最重要的漂移点：

1. 协议文档需要同步 owner 确认口径：`mesh-llm/1` 是当前正式运行协议，shared protocol crate 中的 `mesh-llm/0` 是历史/遗留兼容代码。
2. 运行/发布文档需要同步 owner 确认口径：`v0.60.0` 是节点软件版本最低门槛，代码会拒绝可解析且更低版本的 peer。
3. 多份旧文档仍引用 monolithic 路径，例如 `crates/mesh-llm/src/api`、`src/network`、`src/inference`。
4. `docs/index.html` 仍出现旧式 `llama-server`/`rpc-server` 表达，和当前 embedded Skippy/host runtime 架构不完全一致。
5. `docs/SKIPPY.md`、`LLAMA_STAGE_INTEGRATION_PLAN.md` 更像已完成阶段的计划文档。
6. owner identity 文档仍像 proposal/draft，但代码已有实际 CLI 和验证流程。
7. router 相关设计/benchmark 文档可能落后于当前 `network/router.rs` 和 `mesh-llm-routing`。
8. benchmark 文档里部分命令或结论可能过期，例如 `scripts/openai-smoke.sh` 与当前 `scripts/skippy-openai-smoke.sh` 命名不一致。
9. `context-and-slots-auto.md` 里的代码路径疑似旧路径，但当前实现已在 `runtime/context_planning.rs` 等位置。
10. `docs/plugins/PLAN.md` 仍像路线图，但很多 plugin host/stapler/MCP 能力已经实现。
11. Qwen family 文档包含大量实验/机器/时间敏感内容，不宜直接当当前操作手册。

## 进入第二阶段前必须澄清的问题

第二阶段如果要开始改代码或做架构级二开，建议先确认这些问题：

1. 兼容性文档落地：主线文档是否已经写清 `v0.60.0` 是最低节点软件版本。
2. 协议文档落地：主线文档是否已经写清 `mesh-llm/0` 是历史/遗留兼容代码，不是当前正式运行协议。
3. 架构文档归一：旧 `DESIGN.md` 更新，还是新增/提升 `ARCHITECTURE_CURRENT.md` 为权威？
4. API schema 维护方式：Rust payload、OpenAPI 文档、还是 UI TS 类型，谁是正式合同？
5. owner identity 的产品状态：是默认启用、可选增强，还是未来强制？
6. plugin 协议稳定性：当前 plugin/MCP/stapler 行为是否可以作为二开扩展面。
7. Skippy 文档状态：哪些是当前 runbook，哪些只是历史计划/实验记录。
8. benchmark 口径：哪些数据可以对外引用，哪些必须重新跑。
9. UI 路由和旧 helper：当前 TanStack router 与旧 route helper 是否需要统一。
10. 发布/部署来源：CUDA lane、install flavor、release workflow 的唯一权威文档是哪一份。
11. 接管文档去留：`docs/PD-detach/phase-1/` 是临时分支包，还是后续要提升到 `docs/` 根目录的正式文档。
